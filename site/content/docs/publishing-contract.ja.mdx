---
title: プレビューと出力が一致する仕組み
description: Imposia が表示、印刷、EPUB 作成に使う文書の違いを説明します。
---

画像、表、多数のページを含む文書は、ページ分割に時間がかかります。Imposia は次のバージョンを準備している間も直前の完成済み文書を表示し、未完成のレイアウトを読者に見せません。

## 読者に表示される文書

Core は1つの永続的な canonical iframe を所有します。この iframe には、直近でページ分割に成功した文書が入っています。

React コンポーネントと Viewer は、この iframe を直接表示します。ページを複製したり、2つ目の表示文書を作ったりしません。

## ソース更新時の処理

1. Core が新しいソース用に一時 staging iframe を作成します。
2. 許可されたアセットを読み込み、文書全体をそこでページ分割します。
3. ページ分割に成功すると、Core が canonical iframe の内容を一度に置き換えます。
4. 失敗、キャンセル、より新しい処理への置き換えが起きた場合は、staging iframe を削除し、以前の文書を表示し続けます。

staging iframe をプレビュー、表示、印刷に使うことはありません。

## 出力ごとの生成方法

| 出力 | ソース | 結果 |
| --- | --- | --- |
| プレビュー | Canonical iframe | Viewer が表示する、ページ分割済みの完成ページ |
| 印刷 | `Window.print()` を通じた canonical iframe | ブラウザー標準の印刷フロー。Imposia は PDF バイトを返しません |
| EPUB | 最新のコミット済みセマンティックソース | リフロー型 EPUB 3.3 `Blob`。ページ DOM の固定レイアウトコピーではありません |

`PageDocument.exportEpub()` は extension を再実行せず、固定されたページレイアウトも直列化しません。

## Viewer とネイティブ印刷が一致する理由

まず Viewer で committed generation の2、3ページ目を確認します。次にブラウザーの印刷プレビューで「PDFに保存」を選ぶと、同じ内容とページ順が表示されます。

![Imposia Viewerに表示された同じ2ページ目](/images/publishing-contract/committed-page-viewer.png)

*Viewer の2ページ目には、表の01–22行、繰り返し表ヘッダー、`LEFT / 2` のページ装飾が表示されます。*

![Imposia Viewerに表示された次の表ページとページ装飾](/images/publishing-contract/committed-page-continuation.png)

*Viewer の3ページ目には、続きの23–36行と `RIGHT / 3` のページ装飾が表示されます。*

![PDFに保存を選んだブラウザー印刷プレビューの同じ文書](/images/publishing-contract/native-print-save-pdf.png)

*PDF保存用プレビューにも、上で確認した2、3ページ目と同じ表の内容、ページ順、繰り返し表ヘッダー、ページ装飾が表示されます。*

Imposia は Viewer のページを複製せず、印刷用に別の PDF renderer も作りません。`print()` は Viewer が表示する canonical iframe の `Window.print()` を呼び出します。そのため、文書化された対応範囲では、ページの向き、margin box、繰り返し表ヘッダー、ページ番号、本文の順序が Viewer とブラウザー印刷または PDF 保存結果で一致します。

Viewer と PDF 保存用プレビューの内容が一致するのは、両方の経路が同じ committed generation を参照するためです。Imposia が PDF バイトを直接返すという意味ではありません。プリンタードライバーの余白、拡大縮小、色補正、ブラウザーごとのフォント計測は最終結果に影響する場合があります。

## 外部リソースを制御する

外部リソースは、ホストアプリの `assetResolver` だけを通ります。Core はその経路で作られた Blob URL を所有し、文書の置き換え、失敗、破棄の際に取り消します。

サニタイズ済み入力、resolver 出力、extension には、文書化された制限が適用されます。機能を正確に適用できない場合、Imposia は推測で処理せず、型付き警告を返します。

## 互換性の境界を確認する

- **安定：** ブラウザー ESM API、iframe ライフサイクル、resolver 分離、対応するページメディア、ネイティブ印刷、リフロー型 EPUB。
- **制約付き：** 文書化された表、flex、grid、段組み、参照、named string のサブセット。
- **実験的：** オプトインのページローカル脚注とページフロート。これらの機能は警告を返します。
- **未対応：** Node または CLI レンダリング、サーバー書き出し、PDF バイト、固定レイアウト EPUB、任意の CSS フラグメンテーション、ブラウザー間で同一のページ数。

構造的なページ分割の基準には Chromium を使ってください。Firefox と WebKit も公開 API とライフサイクルをサポートしますが、寸法と改行は異なる場合があります。
